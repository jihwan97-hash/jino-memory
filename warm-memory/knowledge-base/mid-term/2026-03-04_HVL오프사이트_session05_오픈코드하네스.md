## 한줄 요약
하네스는 "모델이 안 되니까 붙이는 목발"이 아니라 같은 모델에서 9할의 성능 차이를 만드는 조직 설계이며, 5분 만에 338줄 베어본을 만들고 실패를 관찰하며 반복 개선하는 것이 하네스 엔지니어링의 본질이다.

## 핵심 포인트
- 하네스 정의: LLM 모델=개인 지능, 하네스=조직 구조/워크플로우 스켈레톤 — 같은 Opus 4.6이어도 하네스에 따라 결과가 달라짐
- 에이전트 세대론: 1세대(워크플로우 기반 LangChain) → 2세대(자율 루프, Claude Code) → 2.5세대(선택지 추상화, 서치 스페이스 축소) → 3세대(하네스를 스스로 만들고 자율 최적화)
- 라이브 빌딩: 5분 만에 338줄 Go 코드 생성(파일 읽기/쓰기/커맨드 실행 포함)
- 첫 번째 실패: list files 호출 시 파일 전체 내용을 읽어버려 컨텍스트 폭발 → "Prompt is too long" 에러
- 핵심 개선: 도구 자체가 아니라 도구의 리턴값을 개선(`.git`, `node_modules` 필터링)
- 시지푸스 vs Claude Code 철학: Claude Code는 방임형("실수하면 다시"), 시지푸스는 선제적 개입("잘못된 행동 전에 먼저 매를 때린다")
- "에이전트를 인간처럼 대하라": 실수 패턴을 귀납적으로 파악 → 시스템이 선제적으로 막아줌
- 토큰 경제학: 하드 필터(.git) → 소프트 필터(캐시) → 다이나믹 필터(컨텍스트 잔여량 기반 프루닝) 3단계
- 해시라인 개선: 공백을 해시에 포함 → 원작자보다 1.5~2배 높은 성능 달성
- TypeScript가 에이전트 하네스의 네이티브 언어로 자리잡는 추세; FriendliAI 3만 토큰/초 이상 처리

## 전문 요약
민웅기(oh-my-opencode 메인테이너) + 김연규(oh-my-opencode 크리에이터) 공동 발표. 에이전트 하네스를 스크래치로 라이브 빌딩하며 개념과 디버깅 과정을 시연.

**하네스란**: MIT 박사 10명도 조직 구조에 따라 결과가 다르듯, LLM의 지능을 어떤 워크플로우 스켈레톤에 담느냐가 성능을 결정. "같은 Opus 4.6인데 Claude Code와 오마이 오픈코드가 다른 결과를 내는 이유 = 하네스 차이".

**에이전트 세대론**: 1세대(LangChain 디터미니스틱 워크플로우) → 2세대(자율 루프, Claude Code류) → 2.5세대(포켓몬 비유: 전투 상황에서 attack/defense 추상화로 서치 스페이스 축소, ~2028 예상) → 3세대(하네스 자체 생성 + 강화학습적 자율 최적화).

**라이브 빌딩**: AMP Code 블로그 튜토리얼 링크를 프롬프트에 붙여넣기 → 오마이 오픈코드 + Opus 4.6으로 5분 만에 338줄 Go 코드 생성. 첫 번째 실패(컨텍스트 폭발) → `.git`, `node_modules` 필터링으로 개선 → 코드베이스 탐색 성공. "이 피드백 루프 자체를 자동화하면 코드 에디팅 에이전트가 되는 것".

**시지푸스 철학**: "에이전트를 인간처럼 대하라" — 실수를 많이 하는 직원의 패턴을 귀납적으로 파악하고, 시스템이 하반(하체)을 막아주면서 상체(침범)가 잘 뚫어주는 구조. 하드→소프트→다이나믹 필터 3단계 토큰 경제학.

**하네스 확장**: 픽셀 디프 2% 미만 탈출 조건의 클론 코딩 에이전트, 코딩 관련 토큰 제거한 엑셀 에이전트, 트레이딩/리서치 에이전트 등 버티컬 맞춤형 하네스 가능. 3월 28일 SF 밋업 예정.
